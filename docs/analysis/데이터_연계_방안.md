# 🔗 실거래가 데이터와 네이버 매물 데이터 연계 방안

> **작성일**: 2026-03-04
> **목적**: 서로 다른 식별자(ID)를 사용하는 공공데이터(실거래)와 네이버 부동산(매물) 데이터를 결합하여 통합 분석 환경 구축

---

## 1. 문제 정의 (식별자 불일치)

현재 수집 중인 두 데이터 소스는 아파트 단지를 식별하는 고유 키(Key)가 다릅니다.

- **실거래가 데이터 (`apt_basic`, `apt_detail`, `apt_trade`)**
  - **Key**: `apt_id` (공공데이터/K-Apt 기반)
  - 위치: `road_address`, `jibun_address`, `latitude`, `longitude`, `admin_dong`
- **네이버 매물 데이터 (`naver_complex`, `naver_listing`)**
  - **Key**: `complex_no` (네이버 고유 단지번호)
  - 위치: `sido_name`, `sgg_name`, `dong_name`, `latitude`, `longitude`

**결론**: 두 테이블을 JOIN 하여 입체적인 분석(실거래가 vs 매물 호가)을 하려면 **`apt_id`와 `complex_no`를 연결해 주는 매핑(Mapping) 테이블**이 필수적입니다.

---

## 2. 데이터 연계(매핑) 프로세스

아래의 3단계 복합 매핑 전략을 통해 정합성을 높이고 자동화를 구현합니다.

### 1단계: 공간 정보(위경도) 기반 1차 매핑
가장 확실한 물리적 위치 데이터를 활용하여 후보군을 좁히고 강제 매핑합니다.

- **방법**: `apt_basic`의 (lat, lon)과 `naver_complex`의 (lat, lon) 간의 거리(Distance) 계산
- **조건**: 두 단지 간 거리가 **100m 이내**일 경우 매핑 후보로 선정
- **특징**: 이름이 달라도(예: "아크로리버파크" vs "반포아크로리버파크") 물리적 위치가 같으면 쉽게 잡힙니다.

### 2단계: 행정동 + 단지명 유사도 기반 2차 매핑
위경도 오차가 있거나 누락된 경우, 텍스트(문자열)를 기반으로 추가 매핑합니다.

- **방법**: 동일 행정동(`admin_dong` ↔ `dong_name`) 내에서 단지명(`apt_name` ↔ `complex_name`)의 **문자열 유사도(Levenshtein Distance, Jaro-Winkler 등)** 계산
- **조건**: 유사도 점수가 80% 이상인 경우 매핑 후보로 확정
- **전처리**: "아파트", "마을", "단지" 등 공통 수식어 제거 후 비교

### 3단계: 매핑 테이블 생성 및 수동 검수 (Human in the Loop)
자동 매핑된 결과를 별도의 테이블에 저장하고, 모호한 케이스는 수동으로 관리합니다.

#### 제안: `complex_mapping` 테이블 스키마
```sql
CREATE TABLE complex_mapping (
    apt_id VARCHAR(50) PRIMARY KEY,       -- 공공/K-Apt 단지 ID
    naver_complex_no VARCHAR(20),         -- 네이버 부동산 단지 ID
    mapping_method VARCHAR(20),           -- 매핑 방식 (예: 'DISTANCE', 'NAME_SIMILARITY', 'MANUAL')
    confidence_score FLOAT,               -- 신뢰도 점수 (0~100)
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 3. 매물 단위(평형) 연계 방안

단지가 매핑되었다 하더라도, **동일한 평형(타입)** 끼리 비교해야 "호가 vs 실거래가" 분석이 성립합니다.

- **실거래 면적**: `apt_trade.exclusive_area` (전용면적, 소수점 존재 예: 84.92)
- **네이버 매물 면적**: `naver_listing.exclusive_area` (전용면적, 정수형 예: 84)

### 면적(평형) 그룹핑 (Binning) 전략
소수점 차이로 인해 1:1 Exact Match가 어려우므로, **오차 범위 ±1㎡ ~ ±2㎡** 수준으로 버킷(Bucket)을 묶어서 비교합니다.

- **예시 SQL (호가와 실거래가 조인)**:
```sql
SELECT 
    b.apt_name,
    t.deal_date,
    t.exclusive_area AS trade_area,
    t.deal_amount AS recent_trade_price,
    l.exclusive_area AS listing_area,
    l.current_price AS current_listing_price
FROM apt_trade t
JOIN complex_mapping m ON t.apt_id = m.apt_id
JOIN naver_listing l ON m.naver_complex_no = l.complex_no
JOIN apt_basic b ON t.apt_id = b.apt_id
WHERE 
    l.is_active = true 
    AND l.trade_type = 'A1' -- 매매
    AND ABS(t.exclusive_area - l.exclusive_area) < 2.0  -- 전용면적 ±2㎡ 이내 매칭
ORDER BY t.deal_date DESC;
```

## 4. 파이프라인 적용 (향후 과제)

**1. 초기 매핑 1회 실행 스크립트 작성** (예: `pipeline/create_mapping.py`)
- Python `haversine` 라이브러리를 사용해 위경도 거리 계산
- `thefuzz` (또는 `fuzzywuzzy`)를 통해 단지명 유사도 측정
- `complex_mapping` DB 테이블 추출

**2. 데일리 업데이트 처리 (파이프라인 통합)**
- 신규 `apt_id`나 신규 `complex_no`가 수집될 때마다 미분류된 단지를 찾아 `complex_mapping`에 자동 INSERT 하는 로직 추가
