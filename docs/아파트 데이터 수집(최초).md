## 1. 개요 (Overview)
본 문서는 국토교통부 공공데이터포털(data.go.kr)의 Open API를 활용하여 수도권(서울, 인천, 경기) 아파트의 단지 정보 및 실거래가(매매/전월세) 데이터를 체계적으로 수집하는 시스템에 대한 요구사항 및 상세 처리 로직을 정의합니다.

### 1.1 배경
- 부동산 시장 분석을 위해 신뢰할 수 있는 공공 데이터의 대량 수집이 필요함.
- API 호출 제한 및 방대한 데이터 양으로 인해 안정적인 자동화 수집 도구가 요구됨.

### 1.2 범위
- **지역**: 서울특별시(11), 인천광역시(28), 경기도(41)
- **대상**: 아파트(Apartment)
- **기간**: 최근 36개월 ~ 현재
- **데이터 종류**: 단지 코드, 기본 정보, 상세 정보, 매매 실거래가, 전월세 실거래가

## 2. 데이터 소스 및 엔드포인트 확인 (Data Source & Endpoints)

모든 API 호출은 `GET` 방식을 사용하며, 공통적으로 서비스 인증키(`serviceKey`)가 필요합니다.

### 2.1 아파트 단지 식별 코드 (Apt Code)
- **API 명**: 국토교통부_공동주택 단지 목록제공 서비스 (`getSidoAptList3`)
- **Full URL**: 
  `https://apis.data.go.kr/1613000/AptListService3/getSidoAptList3`
- **주요 파라미터**:
  - `sidoCode`: 시도 코드 (11, 28, 41)
  - `pageNo`: 페이지 번호
  - `numOfRows`: 페이지 당 목록 수

### 2.2 아파트 단지 정보 (Apt Info)
- **기본 정보 API**: 국토교통부_공동주택 기본 정보 서비스 (`getAphusBassInfoV4`)
  - **Full URL**: `https://apis.data.go.kr/1613000/AptBasisInfoServiceV4/getAphusBassInfoV4`
- **상세 정보 API**: 국토교통부_공동주택 상세 정보 서비스 (`getAphusDtlInfoV4`)
  - **Full URL**: `https://apis.data.go.kr/1613000/AptBasisInfoServiceV4/getAphusDtlInfoV4`
- **주요 파라미터**:
  - `kaptCode`: 단지 식별 코드 (2.1에서 수집된 값 사용)

### 2.3 실거래가 데이터 (Transaction Data)
- **매매 API**: 국토교통부_아파트매매 실거래자료 (`getRTMSDataSvcAptTradeDev`)
  - **Full URL**: `https://apis.data.go.kr/1613000/RTMSDataSvcAptTradeDev/getRTMSDataSvcAptTradeDev`
- **전월세 API**: 국토교통부_아파트 전월세 자료 (`getRTMSDataSvcAptRent`)
  - **Full URL**: `https://apis.data.go.kr/1613000/RTMSDataSvcAptRent/getRTMSDataSvcAptRent`
- **주요 파라미터**:
  - `LAWD_CD`: 지역코드 5자리 (예: 11110 종로구)
  - `DEAL_YMD`: 계약월 (YYYYMM)

## 3. 상세 API 처리 로직 (Detailed Processing Logic)

각 스크립트별 데이터 처리 흐름과 예외 처리 방식은 다음과 같습니다.

### 3.1 단지 코드 수집 (`collect_apt_codes.py`)
1.  **Iterative Request**: 대상 시도 코드 리스트(`['11', '28', '41']`)를 순회합니다.
2.  **Pagination Loop**:
    -   `pageNo`를 1부터 증가시키며 API를 호출합니다 (While Loop).
    -   API 응답의 `body.items`가 비어있거나, 반환된 아이템 수가 `numOfRows`보다 작으면 마지막 페이지로 간주하여 루프를 종료합니다.
3.  **Data Aggregation**: 수집된 모든 리스트를 하나의 리스트로 병합합니다.
4.  **Save**: `apt_code_YYYYMMDD.csv` 파일로 저장합니다.

### 3.2 상세 정보 수집 (`collect_apt_info.py`)
1.  **Pre-check**: 가장 최근의 `apt_code_*.csv` 파일을 로드하여 `kaptCode` 목록을 추출합니다.
2.  **Resume Capability (이어받기)**:
    -   이미 생성된 `apt_basic_info_*.csv` 파일이 있다면 읽어옵니다.
    -   이미 수집된 `kaptCode`를 확인하여 작업 대상 목록에서 제외(Filter)합니다.
3.  **Linear Processing**: 필요한 코드들에 대해 순차적으로:
    -   `getAphusBassInfoV4` 호출 -> 기본 정보 저장.
    -   `getAphusDtlInfoV4` 호출 -> 상세 정보 저장.
    -   **Rate Limiting**: 과도한 요청 방지를 위해 각 호출 사이에 `time.sleep(0.1)` 딜레이를 적용합니다.
4.  **Error Handling**: 특정 단지 정보가 없거나 에러 발생 시 `None`을 반환하고 스킵하며, 프로세스를 중단하지 않습니다.

### 3.3 실거래가 대량 수집 (`collect_trade_rent_data.py`)
1.  **Parameter Preparation**:
    -   `apt_code` 파일의 `bjdCode`(법정동코드, 10자리) 앞 5자리를 잘라 고유한 `LAWD_CD`(지역코드) 목록을 생성합니다.
    -   현재 날짜 기준 과거 36개월치 `DEAL_YMD` 리스트를 생성합니다.
2.  **Nested Loop structure**:
    -   `For Region in LAWD_CODES`:
        -   `For Month in MONTH_LIST`:
            -   API 호출 (매매 & 전월세).
3.  **Internal Pagination**:
    -   거래량이 많은 지역/월(예: 1000건 이상)의 경우 단일 요청으로 데이터가 잘릴 수 있습니다.
    -   응답 데이터 수가 요청한 `numOfRows`와 같으면 `pageNo`를 증가시켜 추가 요청하는 내부 루프가 동작합니다.
4.  **Data Injection**: API 결과에는 지역코드나 년월 정보가 누락될 수 있으므로, 요청 보낼 때 사용한 `LAWD_CD`와 `DEAL_YMD` 값을 결과 데이터에 강제로 주입(Inject)합니다.

## 4. 운영 절차 (Operation Workflow)

### Step 1: 환경 구성
1. Python 3.9+ 및 필수 라이브러리(`pandas`, `requests`, `python-dotenv`, `xmltodict`) 설치.
2. `.env` 파일에 `DATA_API_KEY=Decoded_Key` 설정.

### Step 2: 실행 순서
| 순서 | 스크립트 | 역할 | 예상 소요 시간 |
| :--- | :--- | :--- | :--- |
| **1** | `collect_apt_codes.py` | 단지 코드 마스터 데이터 수집 | 1~2분 |
| **2** | `collect_apt_info.py` | 단지별 상세 스펙 수집 (Resume 가능) | 수 시간 (약 20,000 requests) |
| **3** | `collect_trade_rent_data.py` | 3년치 거래 내역 수집 | 수 시간 (지역수 x 36개월 requests) |

### Step 3: 결과 검증
- `datas/` 폴더 내 CSV 파일 생성 여부 및 용량 확인.
- Excel 등으로 열어 End of File까지 데이터가 정상적으로 기록되었는지 확인.

---

## 5. 제약 사항 및 예외 처리

- **API 트래픽 제한**: 공공데이터포털 정책에 따라 일일 트래픽 초과 시(`LIMITED_NUMBER_OF_SERVICE_REQUESTS_EXCEEDS_ERROR`) 수집이 중단될 수 있습니다. 이 경우 다음 날 이어받기 기능을 활용해야 합니다.
- **데이터 정합성**: API 응답 XML 구조가 비정상적이거나(예: 리스트가 아닌 단일 딕셔너리 반환) `None`일 경우, 스크립트 내부의 `utils.py` 및 파싱 로직에서 이를 리스트로 정규화하거나 예외 처리하여 크래시를 방지합니다.
