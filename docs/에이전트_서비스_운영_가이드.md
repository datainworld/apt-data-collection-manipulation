## 2. 환경 변수 (.env)

| 변수명 | 용도 | 필수 |
|---|---|:---:|
| `NAVER_CLIENT_ID` | 네이버 검색 API Client ID | ✅ |
| `NAVER_CLIENT_SECRET` | 네이버 검색 API Client Secret | ✅ |
| `GOOGLE_API_KEY` | Gemini LLM API 키 | ✅ |
| `POSTGRES_HOST` / `_PORT` / `_USER` / `_PASSWORD` / `_DB` | PostgreSQL 연결 | ✅ |
| `AGENT_MODEL` | LLM 모델명 (기본: `gemini-2.0-flash`) | ❌ |
| `AGENT_TEMPERATURE` | LLM 온도 (기본: `0`) | ❌ |

## 3. 일일 운영 절차

### 3-1. 자동화 파이프라인 (매일 09:00)

```powershell
# 1단계: 뉴스 수집 (약 5초)
uv run python pipeline/collect_news.py

# 2단계: NotebookLM에 뉴스 등록 (약 30~45분)
uv run python pipeline/register_news_to_notebooklm.py
```

> **참고**: Dokploy 스케줄러에 등록할 경우 `cron: 0 9 * * *`으로 설정

### 3-2. WebApp 서버 실행

```powershell
# 로컬 개발 서버
uv run uvicorn webapp.app:app --reload --host 0.0.0.0 --port 8000
```

### 3-3. API 엔드포인트

| 엔드포인트 | 메서드 | 설명 |
|---|---|---|
| `/health` | GET | 헬스 체크 |
| `/api/chat` | POST | 자연어 질문 → Supervisor 그래프 → AI 응답 |
| `/api/news-insight` | GET | 오늘 뉴스 기반 핵심 인사이트 (SQL 데이터 결합) |

#### `/api/chat` 요청 예시

```json
{
  "question": "강남구 최근 아파트 평균 매매가 알려줘",
  "thread_id": "user-123"
}
```

#### `/api/news-insight` 응답 예시

```json
{
  "date": "2026-03-04",
  "news_count": 30,
  "insights": "...(AI 생성 인사이트)...",
  "news_titles": ["뉴스제목1", "뉴스제목2", ...]
}
```

## 4. NotebookLM 관리

### 인증

```powershell
# 최초 1회 또는 토큰 만료 시
uvx --from notebooklm-mcp-cli nlm login
```

### 주요 CLI 명령어

```powershell
# 노트북 목록 조회
uvx --from notebooklm-mcp-cli nlm notebook list

# 노트북 생성
uvx --from notebooklm-mcp-cli nlm notebook create "노트북이름"

# URL 소스 추가 (노트북 ID 사용)
uvx --from notebooklm-mcp-cli nlm source add {노트북ID} --url "https://..."

# 버전 확인
uvx --from notebooklm-mcp-cli nlm --version
```

> **주의**: `source add` 시 노트북 **이름이 아닌 ID**를 사용해야 합니다.

## 5. 에이전트 구성

### Supervisor 라우팅 규칙

| 사용자 질문 유형 | 라우팅 대상 | 예시 |
|---|---|---|
| 데이터 조회/분석 | `sql_agent` | "강남구 아파트 시세", "전세가율 높은 단지" |
| 뉴스/시장 동향 | `notebooklm_agent` | "오늘 부동산 뉴스 요약", "시장 전망" |
| 복합 질문 | sql → notebooklm 순차 | "최근 거래량과 뉴스 트렌드 비교" |

### SQL Agent 안전 규칙

- **SELECT만 허용** (INSERT/UPDATE/DELETE/DROP 차단)
- **LIMIT 100 자동 적용** (미지정 시)
- 쿼리 실행 전 `EXPLAIN` 기반 구문 검증

## 7. 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| `nlm` 명령어 인식 불가 | uv tool로 설치되어 .venv PATH에 없음 | `uvx --from notebooklm-mcp-cli nlm ...` 사용 |
| `source add` 실패 | 노트북 이름 대신 ID 필요 | 노트북 ID(UUID)를 사용 |
| NotebookLM 인증 만료 | Google OAuth 토큰 만료 | `nlm login` 재실행 |
| 뉴스 수집 0건 | 네이버 API 키 미설정 | `.env`에 NAVER_CLIENT_ID/SECRET 확인 |
| SQL Agent DML 시도 | 안전 규칙 동작 | 자동 차단됨 (정상) |

## 8. Dokploy 스케줄러 설정 (서버 자동화)

```bash
# crontab 또는 Dokploy 스케줄러
# 매일 오전 8:50에 뉴스 수집
50 8 * * * cd /app && python pipeline/collect_news.py >> /var/log/news_collect.log 2>&1

# 매일 오전 9:00에 NotebookLM 등록
0 9 * * * cd /app && python pipeline/register_news_to_notebooklm.py >> /var/log/nlm_register.log 2>&1
```

> **서버에서의 NotebookLM 인증**: Dokploy 컨테이너 내에서도 `nlm login`을 한 번 실행하여 인증해야 합니다. 인증 토큰은 `~/.notebooklm/` 디렉토리에 저장되므로, 이 경로를 Docker 볼륨으로 마운트하여 컨테이너 재배포 시에도 유지되도록 설정하세요.
